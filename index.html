Billig lokal förgrening (branching)

Antagligen är Gits mest övertygande egenskap dess flexibla hantering av grenar, som skiljer Git från nästan alla andra versionhanteringssystem (SCM). Grenar i Git är helt annorlunda än i alla andra system som jämförs här, där de flesta syftar till att grenar ska vara en kopia av hela arkivet i en egen mapp.
Git fungerar inte på detta sätt. Git tillåter dig att ha ett flertal lokala grenar som kan vara helt oberoende av varandra och att skapa, slå ihop och ta bort dem tar sekunder.
Detta betyder att du kan göra saker som att
  • skapa en gren för att prova en idé du har, checka in ändringar hur mycket du vill, gå tillbaka till där du var innan, verkställa en patch, gå tillbaka till ditt experiment och till sist slå ihop grenen.
  • ha en gren som alltid innehåller det som är produktionssatt, en annan där du har det som skall testas och flera mindre för dagligt arbete.
  • skapa en ny gren för varje ny funktionalitet du arbetar på så du kan växla fritt mellan dem, och sedan ta bort grenar som redan slagits ihop med din huvudgren.
  • skapa en gren att experimentera i, inse att det inte ledde till något konkret och sedan ta bort den och överge ditt arbete – utan att någon annan har fått se den, trots att du kan ha tryckt ut ändringar i andra grenar under tiden.
Viktigt att förstå är att när du trycker ut dina ändringar till ett externt arkiv så måste du inte också trycka ut alla dina grenar. Du kan välja vilka grenar du vill dela med dig av. Detta brukar ge många möjlighet att experimentera med saker utan att direkt behöva oroa sig för hur och när de ska dela med sig av experimentet till andra.
Du kan komma på sätt att göra detta även med några av de andra systemen, men med följden att du får mer och riskablare extrajobb. Git gör denna process otroligt enkel och det förbättrar arbetsprocessen för de flesta utvecklare när de tagit till sig detta.

Allt är lokalt

Detta är egentligen ganska sant med alla distribuerade SCM:er, men enligt mina erfarenheter så är det extra sant med Git. Det är lite utöver "fetch", "pull" och "push" som kommunicerar på något sätt med något annat än din egen hårddisk.
Detta gör inte bara det mesta operationer snabbare än vad du kanske förväntar dig, utan gör också att du kan arbeta med saker utan en uppkoppling. Det kanske inte låter så viktigt, men jag blir alltid förundrad över hur ofta jag faktiskt jobbar frånkopplat. Att kunna förgrena, slå ihop grenar, checka in ändringar och se historik samtidigt som du sitter i ett tåg eller ett flygplan är väldigt produktivt.
Även i Mercurial så kommunicerar vanliga kommandon som "incoming" och "outgoing" med en server, medan du med Git kan göra en "fetch" och sedan ha all data från servern, vilket ger dig möjligheten att hantera kod som du inte har i dina lokala grenar än, trots att du är frånkopplad.
Detta betyder således att det är mycket enkelt att ha kopior av inte bara dina grenar, utan också grenarna som tillhör de andra som jobbar i samma arkiv utan att det stökar till dina egna ändringar och grenar.

Git är snabbt

Git är snabbt. Alla — till och med de mest inbitna fansen av de andra systemen — brukar ge Git denna titel. Med Git sker alla operationer lokalt, vilket gör Git klart snabbare än SVN och Perforce som båda ansluter till en server för många kommandon. Men även jämfört med de andra distribuerade SCM:arna är Git snabbt.
En förklaring till detta är antagligen att Git skapades just för att arbeta med den gigantiska källkoden till Linuxkärnan, vilket innebär att kravet att effektivt kunna hantera stora arkiv var med från första början. Dessutom är Git skrivet i C, vilket undviker fördröjningar jämfört med andra verktyg som byggts med vissa högnivåspråk. En ytterligare orsak till att git är snabbt är att dess huvudsakliga utvecklare haft snabbhet som en uttalad målsättning.
Nedan följer ett antal hastighetsmätningar gjorda på kopior av Djangos källkodsarkiv. Hastigheten på ett antal operationer mättes för tre olika SCM:er, nämligen Git, Mercurial och Bazaar. Jag provade också med Subversion och tro mig; det är långsammast – ta Bazaars nummer och lägg till nätverksfördröjningar.
Slutresultatet var att för allt utom att lägga till nya filer så var Git snabbast. Detta gäller även riktigt stora incheckningar, där Hg var jämlik. Den incheckning som jag mätte var så stor att du sällan eller aldrig kommer att göra en sådan själv – vanliga incheckningar är mycket snabbare i Git.

Index

Till skillnad från andra system så har Git vad som kallas för "index" eller "förberedelseområde". Det är en mellanliggande plats för dina ändringar där du kan ordna till dem så att de ser ut som du vill innan de checkas in.
Det häftiga med förberedelseområdet är att du kan lägga till ändringar allt eftersom de blir klara och du kan bygga upp din incheckning i steg. Vid incheckningen kommer de ändringar med som du valt, istället för att på gammalt sätt välja ut kompletta filer att checka in.
Detta betyder även att du har möjlighet att välja exakt vilka ändringar i en fil som ska checkas in. Aldrig mer dilemmat med en fil där du gjort två orelaterade ändringar och glömt att checka in dem separat. Med Git kan du placera bara den ena ändringen på förberedelseområdet och checka in, för att sedan förbereda och checka in den andra ändringen. Du kan fördela dina ändringar till precis så många incheckningar som du vill!
Självklart så gör Git det väldigt enkelt att ignorera den här funktionen om du inte vill ha den typen av kontroll — släng bara ett '-a' på ditt incheckningskommando för att lägga till alla filerna i arbetsytan.

Distribuerad

En av de häftigaste finesserna med alla de distribuerade versionshanterarna, Git inkluderad, är att de är distribuerade. Det betyder att istället för att göra en utcheckning av ett arkiv så klonar du arkivet.
En positiv effekt är att även om ni arbetar enligt en centraliserad modell, så kommer alla utvecklare att ha en närmast komplett backup av er centrala server. De kan allihop användas för att återskapa en kopia på er centrala server för ifall centralarkivet skulle gå sönder eller råka ut för datakorruption. Så snart flera utvecklare eller servrar har en uppdaterad klon så har ni en skyddande redundans.
Denna redundans saktar heller inte ned saker och ting något särskilt. I snitt var en utcheckning med SVN endast marginellt snabbare än de distribuerade versionshanterarna. Av alla distribuerade SCM:er jag testat var Git snabbast.

Valfritt arbetsflöde

En av de fantastiska sakerna med Git är att eftersom det är distribuerat och har en så överlägset bra hantering av förgreningar så kan man använda Git utan problem för nästan vilket arbetsflöde man vill.

Subversion-liknande flöde

Ett väldigt vanligt flöde för Git speciellt bland de som konverterat från ett centraliserat SCM, är ett centraliserat arbetsflöde. Git tillåter inte att du trycker upp ändringar om någon annan gjort så sen efter du uppdaterade senast, så en modell där alla trycker upp ändringar till ett och samma ställe fungerar utmärkt.

Flöde med integrationsledare

Ett annat flöde som är vanligt med Git är det med en integrationsledare — en person som checkar in ändringar i det "välsignade" arkivet, och sedan flera utvecklare som klonar det arkivet och arbetar på egna versioner av det, för att sedan be ledaren att hämta in ändringar från dessa. Detta flöde ser man ofta hos fri och öppen mjukvara och på GitHub.

Diktator och löjtnanter

För mer massiva projekt kan man arbeta efter samma flöde som används av Linux-kärnan, där flera personer (så kallade "Löjtnanter") ansvarar för varsin del av projektet. De tar sedan hand om ändringar som rör deras del. Sedan kan en annan ledare ("Diktatorn") ta in de ändringar som löjtnanterna godkänt och sammanfläta i det centrala "välsignade" arkivet.

Om igen, Git är väldigt flexibelt vad gäller olika arbetsflöden, så du kan skapa egna arbetsflöden efter behov.

GitHub

Jag kanske är partisk här, eftersom jag är anställd hos GitHub, men jag lade till denna sektion ändå eftersom så många människor säger att GitHub var ett specifikt skäl till att de valde Git.
GitHub är ett skäl att använda Git för många människor eftersom Github är mer som ett socialt nätverk för kod än en enkel lagringsyta. Folk hittar andra utvecklare eller projekt som liknar vad de själva håller på med och kan enkelt skapa förgreningar och bidra med sina egna ändringar. Detta skapar en stor gemenskap runt Git och runt projekten som använder Git.
Det finns andra tjänster, både för Git och för de andra SCM:er, men få är så användarinriktade eller socialt fokuserade som Github. Och ingen av dem har ens i närheten så många användare. Den sociala aspekten är jätteviktig och kombinationen av Git och GitHub är utmärkt för utveckling av öppen mjukvara.
En gemenskap som GitHub finns helt enkelt inte runt något annan SCM-verktyg.
